# T-1.1.6: Refresh tokens, logout y CORS

**Tipo:** Backend

## Descripción

Configurar las piezas faltantes de seguridad: refresh token strategy, logout, rate limiting y CORS. En el MVP con Auth0, la mayoría de estas son configuraciones del tenant — no código custom.

## Implementación

### Refresh token strategy

Auth0 maneja los refresh tokens nativamente. Configurar en el tenant:

| Setting | Value | Notas |
|---------|-------|-------|
| Refresh Token Rotation | Enabled | Rotate-on-use: cada uso emite un token nuevo |
| Refresh Token Expiration | Absolute: 30 días | Máximo tiempo de vida |
| Refresh Token Reuse Detection | Enabled | Revoca familia si se reutiliza un token rotado |
| Inactivity Timeout | 7 días | Si no se usa en 7 días, expira |

El frontend usa el SDK de Auth0 que maneja refresh automáticamente via `getAccessTokenSilently()`.

### Logout

```go
// Endpoint de logout (redirige a Auth0)
// POST /auth/logout
func (h *AuthHandler) Logout(w http.ResponseWriter, r *http.Request) {
    logoutURL := fmt.Sprintf(
        "https://%s/v2/logout?client_id=%s&returnTo=%s",
        h.config.Auth0Domain,
        h.config.Auth0ClientID,
        url.QueryEscape(h.config.FrontendURL),
    )
    web.JSON(w, http.StatusOK, map[string]string{"logout_url": logoutURL})
}
```

Auth0 invalida la sesión server-side. El frontend limpia tokens locales y redirige.

### Rate limiting

Auth0 incluye rate limiting nativo en sus endpoints de autenticación:
- Login: 10 intentos fallidos → bloqueo temporal (Attack Protection)
- Signup: Configurable desde el dashboard

Para endpoints propios del backend, usar middleware:

```go
// Rate limiter simple para MVP (in-memory, por IP)
func RateLimiter(maxRequests int, window time.Duration) func(http.Handler) http.Handler {
    type client struct {
        count    int
        lastSeen time.Time
    }
    var mu sync.Mutex
    clients := make(map[string]*client)

    return func(next http.Handler) http.Handler {
        return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
            ip := r.RemoteAddr
            mu.Lock()
            c, exists := clients[ip]
            if !exists || time.Since(c.lastSeen) > window {
                clients[ip] = &client{count: 1, lastSeen: time.Now()}
                mu.Unlock()
                next.ServeHTTP(w, r)
                return
            }
            c.count++
            c.lastSeen = time.Now()
            if c.count > maxRequests {
                mu.Unlock()
                web.Error(w, http.StatusTooManyRequests, "rate_limit_exceeded", "Too many requests")
                return
            }
            mu.Unlock()
            next.ServeHTTP(w, r)
        })
    }
}
```

Aplicar en rutas sensibles:
- `/auth/*` → 10 req/min por IP
- `/*/generate` (IA) → 5 req/min por usuario
- `/*/chat` (IA) → 10 req/min por usuario

### CORS

```go
// Configurar en cmd/main.go
cors := middleware.CORS(middleware.CORSConfig{
    AllowedOrigins:   strings.Split(config.AllowedOrigins, ","), // env var
    AllowedMethods:   []string{"GET", "POST", "PUT", "PATCH", "DELETE", "OPTIONS"},
    AllowedHeaders:   []string{"Authorization", "Content-Type"},
    AllowCredentials: true,
    MaxAge:           86400, // 24h preflight cache
})
```

Env vars:
- `ALLOWED_ORIGINS`: comma-separated (ej: `http://localhost:3000,https://alizia.educabot.com`)

## Notas

- Auth0 Attack Protection cubre brute force en login — no requiere implementación custom
- El rate limiter in-memory es suficiente para MVP con un solo pod; en producción escalar con Redis
- CORS debe configurarse antes de ir a staging — sin esto el frontend no puede conectar
- El logout endpoint es un redirect helper; la invalidación real la hace Auth0
- Para el futuro auth-service custom, el refresh token y rate limiting se implementan como en el archivo de arquitectura del archive (refresh_tokens table, login_attempts table, bcrypt)
