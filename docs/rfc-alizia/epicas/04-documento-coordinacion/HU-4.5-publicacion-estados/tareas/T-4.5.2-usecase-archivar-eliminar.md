# T-4.5.2: Usecases AutoTransition, ArchiveDocument y DeleteDocument

**Historia:** HU-4.5 — Publicacion y estados
**Tipo:** Backend
**Estado:** ⬜ Pendiente

---

## Descripcion

Implementar tres operaciones para completar el ciclo de vida de los documentos de coordinacion:

1. **AutoTransitionToInProgress**: transiciona automaticamente un documento de `pending` a `in_progress` cuando el coordinador interactua por primera vez (P1).
2. **ArchiveDocument**: archiva un documento desde `in_progress` o `published` (P2).
3. **DeleteDocument**: elimina permanentemente un documento, pero solo si esta en estado `pending`.

### AutoTransitionToInProgress (P1 — Decision)

Transicion automatica: `pending` -> `in_progress` al primer edit/click/interaccion.

Esta logica se invoca **antes de cualquier operacion de edicion** (PATCH secciones, chat, etc.):

```go
func (uc *AutoTransitionUsecase) EnsureInProgress(ctx context.Context, orgID uint, docID uint) error {
    doc, err := uc.docRepo.GetByID(ctx, orgID, docID)
    if err != nil {
        return err
    }

    if doc.Status == "pending" {
        return uc.docRepo.UpdateStatus(ctx, orgID, docID, "in_progress")
    }

    return nil // ya esta en in_progress, published, o archived
}
```

**Donde se invoca:**
- Antes de `PATCH /coordination-documents/:id` (editar secciones)
- Antes de `POST /coordination-documents/:id/chat` (chat con Alizia)
- Antes de `POST /coordination-documents/:id/generate` (generar contenido)
- Antes de `PATCH /coord-doc-classes/:id` (editar clase)

No se expone como endpoint separado — es logica interna que los handlers llaman automaticamente.

### ArchiveDocument (P2 — Decision)

Transicion: `in_progress` -> `archived` o `published` -> `archived`

```go
func (uc *ArchiveDocumentUsecase) Execute(ctx context.Context, orgID uint, docID uint) error {
    doc, err := uc.docRepo.GetByID(ctx, orgID, docID)
    if err != nil {
        return err
    }

    if doc.Status == "pending" {
        return NewValidationError("documentos pendientes deben eliminarse, no archivarse")
    }

    if doc.Status == "archived" {
        return NewValidationError("el documento ya esta archivado")
    }

    return uc.docRepo.UpdateStatus(ctx, orgID, docID, "archived")
}
```

**Endpoint**: `PATCH /api/v1/coordination-documents/:id` con body `{"status": "archived"}`.

**Response exitosa**: `200 OK` con el documento actualizado.

**Notas:**
- No hay restriccion de cantidad por area: multiples documentos pueden coexistir en cualquier estado (P2)
- Archivar no elimina datos — es un soft-transition, no un soft-delete
- Un documento archivado no es visible para docentes

### DeleteDocument

Solo permite eliminar documentos en estado `pending`. Realiza un hard delete con CASCADE en todas las tablas asociadas.

```go
func (uc *DeleteDocumentUsecase) Execute(ctx context.Context, orgID uint, docID uint) error {
    doc, err := uc.docRepo.GetByID(ctx, orgID, docID)
    if err != nil {
        return err
    }

    if doc.Status != "pending" {
        return NewForbiddenError("solo documentos pendientes pueden ser eliminados")
    }

    return uc.docRepo.HardDelete(ctx, orgID, docID)
}
```

**Tablas afectadas por CASCADE**:

```
coordination_documents (eliminado)
  ├── coord_doc_topics (CASCADE)
  ├── coordination_document_subjects (CASCADE)
  │     ├── coord_doc_subject_topics (CASCADE)
  │     └── coord_doc_classes (CASCADE)
  │           └── coord_doc_class_topics (CASCADE)
  ├── coord_doc_chat_messages (CASCADE)
  ├── coord_doc_snapshots (CASCADE)
  └── (sections en JSONB, se elimina con el registro principal)
```

**Endpoint**: `DELETE /api/v1/coordination-documents/:id`

**Responses**:

- `204 No Content` — eliminacion exitosa
- `403 Forbidden` — documento no esta en pending
- `404 Not Found` — documento no existe

### Registro en router

```go
docs := v1.Group("/coordination-documents")
docs.PATCH("/:id", handler.PatchCoordinationDocument, RequireRole("coordinator"))
docs.DELETE("/:id", handler.DeleteCoordinationDocument, RequireRole("coordinator"))
docs.POST("/:id/publish", handler.PublishCoordinationDocument, RequireRole("coordinator"))
```

### Logica en el handler PATCH para estados

```go
func (h *Handler) PatchCoordinationDocument(c echo.Context) error {
    // ... parsear request ...

    if req.Status != nil {
        switch *req.Status {
        case "archived":
            return h.archiveUsecase.Execute(ctx, orgID, docID)
        default:
            return echo.NewHTTPError(422, "invalid status transition via PATCH")
        }
    }

    // Auto-transicion pending → in_progress antes de editar
    h.autoTransitionUsecase.EnsureInProgress(ctx, orgID, docID)

    // ... otros campos editables (secciones) ...
    // Si documento esta published, solo permitir editar secciones (no clases) + warning
}
```

## Notas

- La transicion `pending -> in_progress` es automatica e invisible para el usuario (P1)
- No hay camino de retorno: no se puede "despublicar" ni "desarchivar"
- La publicacion tiene su propio endpoint y usecase separado (T-4.5.1) con validaciones
- `archived` no implica eliminacion — los datos persisten para referencia historica
