# T-4.5.2: Usecases ArchiveDocument y DeleteDocument

**Historia:** HU-4.5 — Publicacion y estados
**Tipo:** Backend
**Estado:** ⬜ Pendiente

---

## Descripcion

Implementar dos usecases para completar el ciclo de vida de los documentos de coordinacion:

1. **ArchiveDocument**: transiciona un documento publicado a archivado. No requiere validaciones adicionales.
2. **DeleteDocument**: elimina permanentemente un documento, pero solo si esta en estado borrador. Los documentos publicados o archivados no se pueden eliminar.

## Implementacion

### Ubicacion

`src/core/usecases/coordination/archive_document.go`
`src/core/usecases/coordination/delete_document.go`

---

### ArchiveDocument

Transicion: `published` -> `archived`

```go
func (uc *ArchiveDocumentUsecase) Execute(ctx context.Context, docID int) error {
    doc, err := uc.docRepo.GetByID(ctx, docID)
    if err != nil {
        return err
    }

    if doc.Status != "published" {
        return NewValidationError("solo documentos publicados pueden ser archivados")
    }

    return uc.docRepo.UpdateStatus(ctx, docID, "archived")
}
```

**Endpoint**: Se invoca desde `PATCH /api/v1/coordination-documents/:id` cuando el body incluye `{"status": "archived"}`.

**Response exitosa**: `200 OK` con el documento actualizado.

---

### DeleteDocument

Solo permite eliminar documentos en estado `draft`. Realiza un hard delete con CASCADE en todas las tablas de junction asociadas.

```go
func (uc *DeleteDocumentUsecase) Execute(ctx context.Context, docID int) error {
    doc, err := uc.docRepo.GetByID(ctx, docID)
    if err != nil {
        return err
    }

    if doc.Status != "draft" {
        return NewForbiddenError("solo documentos en borrador pueden ser eliminados")
    }

    return uc.docRepo.HardDelete(ctx, docID)
}
```

**Tablas afectadas por CASCADE**:

```
coordination_documents (eliminado)
  ├── coord_doc_topics (CASCADE)
  ├── coord_doc_subjects (CASCADE)
  │     ├── coord_doc_subject_topics (CASCADE)
  │     └── coord_doc_classes (CASCADE)
  │           └── coord_doc_class_topics (CASCADE)
  └── (sections en JSONB, se elimina con el registro principal)
```

**Endpoint**: `DELETE /api/v1/coordination-documents/:id`

**Responses**:

- `204 No Content` — eliminacion exitosa
- `403 Forbidden` — documento no esta en draft

```json
{
  "error": "forbidden",
  "message": "Solo documentos en borrador pueden ser eliminados"
}
```

- `404 Not Found` — documento no existe

### Registro en router

En `mapping.go`, los endpoints ya existentes manejan estos casos:

```go
// PATCH para cambio de estado (publicar, archivar)
docs.PATCH("/:id", handler.PatchCoordinationDocument, RequireRole("coordinator"))

// DELETE para eliminacion
docs.DELETE("/:id", handler.DeleteCoordinationDocument, RequireRole("coordinator"))
```

### Logica en el handler PATCH para estados

```go
func (h *Handler) PatchCoordinationDocument(c echo.Context) error {
    // ... parsear request ...

    if req.Status != nil {
        switch *req.Status {
        case "published":
            return h.publishUsecase.Execute(ctx, docID)
        case "archived":
            return h.archiveUsecase.Execute(ctx, docID)
        default:
            return echo.NewHTTPError(422, "invalid status transition")
        }
    }

    // ... otros campos editables ...
}
```

## Notas

- Las foreign keys en las tablas de junction deben tener `ON DELETE CASCADE` definido en las migraciones para que el hard delete funcione correctamente.
- No se implementa soft delete en esta POC para mantener la simplicidad.
- La transicion de estados valida es unidireccional: `draft` -> `published` -> `archived`. No hay camino de retorno.
- Un documento archivado es de solo lectura, igual que uno publicado.
